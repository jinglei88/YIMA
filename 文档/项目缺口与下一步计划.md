## 当前真实缺口（按优先级）

1. 缺少自动化 CI。
   - 现状：已有 `scripts/run_regression.py`、`scripts/run_docs_regression.py`、`scripts/run_v1_contract_regression.py`，但无自动托管执行入口。
   - 风险：多人协作下，回归只能靠手工触发，容易漏测后合并。

2. 编辑器主文件仍过大，变更风险高。
   - 现状：`易码编辑器.py` 仍约 5720 行，UI/补全/语义/导出配置耦合在同文件。
   - 风险：任意功能修改都可能触发连锁影响，审查和定位成本高。

3. 打包链路回归覆盖不完整。
   - 现状：主回归覆盖了运行语义和编辑器逻辑，但没有专门覆盖“导出配置校验 + 依赖收集 + 命令拼装”的稳定性边界。
   - 风险：打包路径、图标、模块依赖变动时，可能只在手工打包阶段暴露问题。

4. 运行产物治理还不彻底。
   - 现状：已有隔离执行修复，但仓库中仍混有示例数据库/导出文件等长期演化数据。
   - 风险：示例数据与运行产物界限不清，后续变更难判断“应该提交”还是“应该忽略”。

## 下一步可执行计划（建议顺序）

1. 建立 CI（优先级 P0）
   - 目标：每次提交自动执行三类回归脚本。
   - 验收：CI 失败可阻断合并，且日志能直达失败脚本。

2. 编辑器逻辑二次拆分（优先级 P0）
   - 目标：继续把补全上下文提取/签名推断/模块成员推断迁移到 `yima/editor_logic_core.py`。
   - 验收：`易码编辑器.py` 降到 4500 行以内，`run_editor_logic_regression.py` 全绿。

3. 新增打包专项回归（优先级 P1）
   - 目标：增加 `scripts/run_packaging_regression.py`，覆盖导出入口解析、模块依赖图构建、非法配置拦截。
   - 验收：核心打包逻辑可在无 GUI 条件下回归，失败信息可定位到具体配置项。

4. 样例数据与运行产物分层（优先级 P1）
   - 目标：明确“演示基线数据”和“运行生成数据”目录边界。
   - 验收：根目录不再产生新的临时 DB/JSON，回归执行后 `git status` 仅出现代码变更。
