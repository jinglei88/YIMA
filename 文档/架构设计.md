# 易码 — 技术架构设计文档（大白话版）

## 一、系统架构总览

易码采用了经典的解释器架构（Lexer -> Parser -> AST -> Interpreter），但在**词法分析**和**语法分析**阶段，专门针对大白话结构进行了深度定制。

```
┌──────────────────────────────────────────┐
│              用户输入层                    │
│  ┌──────────┐    ┌────────────────────┐  │
│  │ REPL 模式 │    │  .ym 文件执行模式   │  │
│  └─────┬────┘    └────────┬───────────┘  │
│        └────────┬─────────┘              │
└─────────────────┼────────────────────────┘
                  ▼
┌──────────────────────────────────────────┐
│         词法分析器 (Lexer)                │
│  将中文短句拆分为有意义的 Token           │
│  • 识别多字关键字（如"设定为"、"的话"）     │
│  • 忽略无逻辑意义的缩进（换行仅作语句分隔）│
│  • 将"加上"、"减去"映射为数学算符          │
└─────────────────┬────────────────────────┘
                  ▼
┌──────────────────────────────────────────┐
│         语法分析器 (Parser)               │
│  基于自然语言句式的解析 (Natural Parsing) │
│  • 匹配句型模板（"把" + 变量 + "设定为" + 表达式） │
│  • 寻找显式的结束标记（"结束"、"学完了"）   │
│  • 生成标准化的抽象语法树 (AST)            │
└─────────────────┬────────────────────────┘
                  ▼
┌──────────────────────────────────────────┐
│         抽象语法树 (AST)                  │
│  (内部结构恢复为标准的计算机逻辑树)       │
└─────────────────┬────────────────────────┘
                  ▼
┌──────────────────────────────────────────┐
│          解释器 (Interpreter)             │
│  通过访问者模式遍历执行                   │
└──────────────────────────────────────────┘
```

## 二、大白话解析核心设计

### 2.1 词法分析 (Lexer) 重大变更

传统的编程语言 Lexer 按空格或固定符号切分。易码的 Lexer 必须能识别**多字中文词组**：

```python
class Token类型(Enum):
    # 动作与状态
    关键字_把, 关键字_设定为, 关键字_显示, 关键字_询问
    
    # 逻辑结构
    关键字_如果, 关键字_的话, 关键字_否则如果, 关键字_不然的话
    关键字_当, 关键字_的时候, 关键字_重复, 关键字_次
    关键字_结束, 关键字_学完了
    
    # 操作符对应
    算符_加上, 算符_减去, 算符_乘以, 算符_除以, 算符_拼接
    比较_大于, 比较_小于, 比较_等于, 比较_不等于, 比较_至少是, 比较_最多是
```

**解析词组**：当遇到"设"，Lexer会往下看是否是"设定为"，如果不是则报错，如果是则生成 `Token(设定为)`。
不再通过缩进计算层级，去掉了 `Indent` 和 `Dedent` Token。

### 2.2 语法分析 (Parser) 句式匹配

Parser 的逻辑不再是传统的递归下降找分号，而是**匹配中文句式模板**：

**1. 赋值句式解析**
```python
def _解析赋值语句():
    匹配(Token类型.关键字_把)
    变量名 = 获取匹配(Token类型.标识符).值
    匹配(Token类型.关键字_设定为)
    表达式 = _解析表达式()
    return 变量赋值节点(变量名, 表达式)
```

**2. 条件句式解析**
```python
def _解析如果语句():
    匹配(Token类型.关键字_如果)
    条件 = _解析表达式()
    匹配(Token类型.关键字_的话)
    
    真分支_语句块 = []
    # 循环读取语句，直到遇到"否则如果"、"不然的话"或"结束"
```

**3. 表达式中的中文化算符翻译**
Parser中的 Pratt Parsing 算法，将中文词组映射为标准算符优先级：
- `乘以`、`除以` -> 优先级 4
- `加上`、`减去`、`拼接` -> 优先级 5
- `大于`、`等于`、`至少是` -> 优先级 6

## 三、抽象语法树 (AST)
尽管外部语言是大白话，但内部的 AST 节点设计依然遵循经典的抽象结构：

```
AST节点 (基类)
├── 表达式节点
│   ├── 数字字面量节点     {值: number}
│   ├── 文本字面量节点     {值: string}
│   ├── 变量访问节点       {名称: string}
│   ├── 二元运算节点       {左: 表达式, 运算符: string, 右: 表达式}
│   └── 函数调用节点       {函数: 表达式, 参数: [表达式]}
└── 语句节点
    ├── 显示语句节点       {表达式: 表达式}
    ├── 变量赋值节点       {名称: string, 值: 表达式} # 对应"把...设定为..."
    ├── 如果语句节点       {条件: 表达式, 真分支: [语句], 假分支: [语句]}
    ├── 循环语句节点       {循环体: [语句], 循环类型: 次数循环/条件循环/遍历循环}
    ├── 函数定义节点       {名称: string, 参数: [string], 函数体: [语句]}
    └── 返回语句节点       {值: 表达式} # 对应"交出..."
```

## 四、运行时与类型系统

运行时(Runtime)没有区别，但在发生错误时，`解释器.py` 和 `错误.py` 会将内部的类型不匹配转化为大白话报错：

```python
# 传统的报错: TypeError: unsupported operand type(s) for +: 'int' and 'str'
# 易码的报错:
"❌ 哎呀，出错了（第 3 行）：
 你不能把 [数字] 和 [文本] 【加上】，如果你想连起来，请用【拼接】哦"
```

## 五、架构优势与挑战

### 优势
1. **完全无歧义**：明确的主谓宾结构（把 A 设定为 B），让 Parser 写起来其实比 C 语言更简单，不需要猜测 `a = b` 是赋值还是比较的可能写错。
2. **零基础友好**：用户通过阅读句子就能自查逻辑错误，不再受限于对符号系统的陌生感。

### 挑战
1. **词组匹配性能**：Token 种类变多，多字词语匹配需要向前看(lookahead)多个字符，稍微增加了一点点开销（但对于教学级语言完全可忽略）。
2. **函数调用的自然语言化**："教电脑..."的定义很自然，但由于中文表达调用的方式太多变，最终在调用时保留了 `函数名(参数)` 的形式，没有做到 100% 取消括号。
