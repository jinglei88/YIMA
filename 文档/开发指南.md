## 三、开发约定与核心原则

### 1. 忘掉 Python，拥抱大白话
在开发 `yima/语法分析.py` 时，**绝对不要**把英文符号和语法生搬硬套给用户。
如果有新功能提议（比如 try-catch），我们必须要先想："如果是没有学过编程的小学同学，他会怎么说话？"
比如：
- 错误示范：`尝试...如果出错...`
- 好的示范：`如果 运行不出错 的话... 但要是弄砸了就...`

### 2. "啰嗦"的 Token
易码的 Token 名字会比传统语言长。这是正常的。

| 传统 Token | 易码 Token |
|------------|------------|
| `ASSIGN` (=) | `关键字_把`, `关键字_设定为` |
| `IF` | `关键字_如果`, `关键字_的话` |
| `RBRACE` (}) | `关键字_结束` / `关键字_学完了`|

### 3. 错误提示必须卖萌/口语化
任何 `yima/错误.py` 里抛出的异常，必须包含行号，并且必须是可以大声读出来的正常句子。

```python
# 测试和开发中：
raise 易码语法报错("我看到了【本来】，但却没有找到之前的【如果】，这可不对劲哦。")
```

## 四、如何添加一个新功能（示例：添加逻辑"反过来显示"）

假设我们想实现 `not` 逻辑，对应的中文是 `反过来说`。我们需要按顺序改四个地方：

1. **`词法分析.py`**
   - 在关键字列表里加上 `"反过来显示"`。
   - 分词器遇到这段中文字符串就会吐出 `Token(逻辑_反过来说)`。

2. **`语法分析.py`**
   - 找到 `_解析前缀表达式` 方法。
   - 补充逻辑：`如果 token 是 Token(逻辑_反过来说)`，就返回一个 `一元逻辑节点("反过来显示", 右侧表达式)`。

3. **`解释器.py`**
   - 找到 `_执行_一元逻辑节点`。
   - 写业务逻辑：`如果 操作符 是 "反过来显示"，就 return not(计算出右边的值)`。

4. **`测试/test_语法.py`**
   - 补充代码：`分析器解析("如果 反过来显示 下雨 的话 显示 1")` 确保不报错。

## 五、测试规范

我们对"分词结果"和"AST生成"卡得很严。必须大量写类似下面的测试用例：

```python

    # 期望用户写大白话
    代码 = '把 年龄 设定为 18'
    tokens = 分词(代码)
    assert tokens[0].类型 == Token类型.关键字_把
    assert tokens[1].类型 == Token类型.标识符
    assert tokens[1].值 == '年龄'
    assert tokens[2].类型 == Token类型.关键字_设定为
    assert tokens[3].类型 == Token类型.数字

```
